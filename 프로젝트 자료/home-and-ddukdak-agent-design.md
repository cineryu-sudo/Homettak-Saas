## 1. 작업 컨텍스트

### 1.1 배경 및 목적

홈앤뚝딱은 가정용 설치/수리 서비스를 제공하며, 고객 문의 접수부터 기사 배정, 설치 완료, 입금 확인까지의 프로세스를 운영 중이다. 현재 대부분의 작업이 수동으로 진행되어 다음과 같은 문제가 발생한다:

- 카카오톡/네이버톡톡 문의에서 고객 정보를 일일이 추출하여 엑셀에 입력
- 기사 배정 시 주소를 보고 담당 지역 기사를 수동 매칭
- 기사에게 고객 정보를 카톡/문자로 직접 작성하여 전달
- 반복 작업으로 인한 시간 소모 및 휴먼 에러 발생

**목적**: 문의 접수 → 정보 정리 → 기사 배정 → 정보 전달 프로세스를 자동화하여 운영 효율성을 높인다.

### 1.2 범위

**자동화 대상 (In Scope)**:
- 카카오톡/네이버톡톡 문의 내용 파싱 및 정보 추출
- 고객 정보 엑셀 자동 기록
- 주소 기반 기사 자동 배정
- 기사용 안내 메시지 자동 생성

**자동화 제외 (Out of Scope)**:
- 전화 문의 자동 응대 (수동 입력 후 동일 흐름 적용)
- 실제 카카오톡/문자 발송 API 연동 (메시지 생성까지만)
- 실시간 입금 확인 자동화 (4순위, 추후 확장)

### 1.3 입출력 정의

**입력**:
| 입력 유형 | 형식 | 설명 |
|----------|------|------|
| 고객 문의 내용 | 텍스트 (복사/붙여넣기) | 카톡/톡톡 대화 내용 |
| 기사 명단 | 엑셀 파일 | 기사명, 연락처, 담당지역 |
| 고객 DB | 엑셀 파일 | 기존 고객 정보 (있는 경우) |

**출력**:
| 출력 유형 | 형식 | 설명 |
|----------|------|------|
| 정리된 고객 정보 | 엑셀 행 추가 또는 JSON | 이름, 연락처, 주소, 서비스 내용, 희망일시 |
| 배정 결과 | JSON/텍스트 | 담당 기사 정보 |
| 기사 안내 메시지 | 텍스트 | 복사하여 카톡/문자로 전송 가능한 형태 |

### 1.4 주요 제약조건

- 기사 정보와 고객 정보는 엑셀 파일로 관리 (DB 없음)
- 메시지 발송은 수동 (API 연동 없음, 메시지 생성까지만 자동화)
- 기사 배정은 지역(주소) 기반 단일 기준
- 전화 문의는 운영자가 직접 정보를 입력한 후 자동화 흐름에 합류

### 1.5 용어 정의

| 용어 | 정의 |
|------|------|
| 문의 접수 | 고객이 카톡/톡톡/전화로 서비스를 요청하는 행위 |
| 기사 | 현장 방문하여 설치/수리를 수행하는 담당자 |
| 배정 | 고객의 주소를 기준으로 담당 기사를 지정하는 행위 |
| 안내 메시지 | 기사에게 전달하는 고객 정보 및 작업 내용 요약 |

### 2.2 단계별 상세 정의

#### STEP 1: 고객 정보 추출

| 항목 | 내용 |
|------|------|
| **처리 주체** | LLM (에이전트) |
| **입력** | 카톡/톡톡 대화 텍스트 또는 수동 입력 내용 |
| **수행 작업** | 대화에서 고객명, 연락처, 주소, 서비스 요청 내용, 희망 일시 추출 |
| **출력** | 구조화된 고객 정보 (JSON) |
| **성공 기준** | 필수 필드(이름, 연락처, 주소, 서비스내용) 모두 추출 |
| **검증 방법** | 스키마 검증 + LLM 자기 검증 (추출 신뢰도) |
| **실패 처리** | 에스컬레이션 - 누락 항목 명시하여 운영자에게 확인 요청 |

**추출 대상 필드**:
```
{
  "customer_name": "필수",
  "phone": "필수",
  "address": "필수",
  "address_region": "자동 파생 (시/군/구)",
  "service_type": "필수 (설치/수리/점검 등)",
  "service_detail": "선택 (구체적 요청사항)",
  "preferred_date": "선택",
  "preferred_time": "선택",
  "notes": "선택 (기타 특이사항)"
}
```

**LLM 판단 영역**:
- 비정형 대화에서 정보 식별 및 추출
- 동일 정보의 중복/수정 내용 판단 (예: 주소를 두 번 말한 경우 최신 것 사용)
- 서비스 유형 분류 (설치/수리/점검/기타)
- 추출 신뢰도 자기 평가

---

#### STEP 2: 고객 정보 기록

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 |
| **입력** | STEP 1의 구조화된 고객 정보 (JSON) |
| **수행 작업** | 엑셀 파일에 새 행 추가 |
| **출력** | 업데이트된 엑셀 파일, 기록 완료 확인 |
| **성공 기준** | 엑셀에 정상 기록, 중복 체크 통과 |
| **검증 방법** | 규칙 기반 (행 추가 확인, 필수 컬럼 존재) |
| **실패 처리** | 자동 재시도 (최대 2회) → 실패 시 에스컬레이션 |

---

#### STEP 3: 기사 배정

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 (+ LLM 보조 판단) |
| **입력** | 고객 주소의 지역 정보, 기사 명단 엑셀 |
| **수행 작업** | 주소에서 지역(시/군/구) 추출 → 담당 기사 매칭 |
| **출력** | 배정된 기사 정보 (이름, 연락처) |
| **성공 기준** | 1명 이상의 담당 기사 매칭 |
| **검증 방법** | 규칙 기반 (매칭 결과 존재 여부) |
| **실패 처리** | 에스컬레이션 - 담당 기사 없음 알림, 수동 배정 요청 |

**스크립트 처리**:
- 주소 문자열에서 시/군/구 파싱
- 기사 명단에서 담당지역 컬럼과 매칭

**LLM 보조 판단** (필요 시):
- 주소 형식이 비정형일 때 지역 추론
- 복수 기사 매칭 시 우선순위 판단 (현재는 첫 번째 선택)

---

#### STEP 4: 기사 안내 메시지 생성

| 항목 | 내용 |
|------|------|
| **처리 주체** | LLM (에이전트) |
| **입력** | 고객 정보, 배정된 기사 정보 |
| **수행 작업** | 기사에게 전달할 안내 메시지 생성 |
| **출력** | 복사 가능한 텍스트 메시지 |
| **성공 기준** | 필수 정보 포함, 간결하고 명확한 형식 |
| **검증 방법** | LLM 자기 검증 (필수 정보 포함 여부 체크리스트) |
| **실패 처리** | 자동 재시도 (최대 1회) |

**메시지 포함 필수 항목**:
- 고객명
- 연락처
- 주소
- 서비스 내용
- 희망 일시 (있는 경우)
- 특이사항 (있는 경우)

**메시지 템플릿 예시**:
```
[홈앤뚝딱 작업 배정]

▶ 고객명: {customer_name}
▶ 연락처: {phone}
▶ 주소: {address}
▶ 서비스: {service_type} - {service_detail}
▶ 희망일시: {preferred_date} {preferred_time}
▶ 비고: {notes}

확인 후 고객님께 연락 부탁드립니다.
```

### 2.3 분기 조건 요약

| 분기점 | 조건 | 경로 |
|--------|------|------|
| 문의 채널 | 카톡/톡톡 vs 전화 | 카톡→자동 파싱 / 전화→수동 입력 후 합류 |
| 정보 완성도 | 필수 필드 충족 여부 | 충족→다음 단계 / 미충족→에스컬레이션 |
| 기사 매칭 | 담당 기사 존재 여부 | 존재→메시지 생성 / 없음→에스컬레이션 |

### 2.4 상태 전이

```
[대기] → [정보추출중] → [정보검증] → [기록완료] → [배정중] → [배정완료] → [메시지생성] → [완료]
                           ↓                          ↓
                    [정보부족-대기]              [배정불가-대기]
```

### 3.3 에이전트 구조

- 워크플로우가 선형적이고 단순함 (4단계)
- 각 스킬의 지침이 짧아 컨텍스트 윈도우 부담 없음
- 역할 간 복잡한 조율 불필요

### 3.4 작업 단계별 처리 방식

| 단계 | 처리 방식 | 담당 스킬 |
|------|----------|----------|
| STEP 1: 정보 추출 | **에이전트 판단** | customer-info-extractor |
| STEP 2: 엑셀 기록 | 스크립트 | excel-handler |
| STEP 3: 기사 배정 | 스크립트 (+ 에이전트 보조) | engineer-matcher |

### 3.5 스킬 파일 목록

| 스킬명 | 역할 | 트리거 조건 |
|--------|------|------------|
| **customer-info-extractor** | 대화에서 고객 정보 추출 및 구조화 | 카톡/톡톡 대화 텍스트가 입력되었을 때 |
| **excel-handler** | 엑셀 파일 읽기/쓰기/중복체크 | 고객 정보 저장 또는 기사 명단 조회 시 |
| **engineer-matcher** | 주소 파싱 및 기사 매칭 | 고객 정보 기록 완료 후 |
| **message-generator** | 기사용 안내 메시지 생성 | 기사 배정 완료 후 |

### 3.6 스크립트 파일 목록

| 스크립트 | 위치 | 역할 |
|----------|------|------|
| `validate_customer_info.py` | customer-info-extractor/scripts | 추출된 정보의 스키마 검증 |
| `read_excel.py` | excel-handler/scripts | 엑셀 파일 읽기 |
| `write_excel.py` | excel-handler/scripts | 엑셀 파일에 행 추가 |
| `find_duplicate.py` | excel-handler/scripts | 연락처 기준 중복 고객 체크 |
| `parse_address.py` | engineer-matcher/scripts | 주소에서 시/군/구 추출 |
| `match_engineer.py` | engineer-matcher/scripts | 지역-기사 매칭 수행 |

### 3.7 데이터 전달 방식

| 구간 | 방식 | 형식 |
|------|------|------|
| 입력 → STEP 1 | 프롬프트 인라인 | 텍스트 (대화 내용) |
| STEP 1 → STEP 2 | 파일 기반 | `/output/temp/customer_info.json` |
| STEP 2 → STEP 3 | 프롬프트 인라인 | 고객 주소 문자열 |
| STEP 3 → STEP 4 | 프롬프트 인라인 | 기사 정보 + 고객 정보 |
| STEP 4 → 출력 | 파일 기반 | `/output/messages/{timestamp}.txt` |

### 3.8 주요 산출물 파일 형식

| phone | string | 연락처 |
